16 ### Concept
17 
18 **jm_Scheduler** schedules repeated and intervaled routines like the JavaScript `setInterval()` function does,
19 but with some improvements:
20 
21 - By default, **jm_Scheduler** starts immediately the routine and repeats it periodically.
22 - The first execution can be differed.
23 - The repeated executions can be voided.
24 - The interval between executions can be dynamically changed.
25 - The scheduled routine function can be dynamically changed.
26 - The scheduled routine can be stopped and later restarted.
27 
28 **jm_Scheduler** doesn't schedule like the official [**Scheduler** Library for Arduino DUE and ZERO](https://www.arduino.cc/en/Reference/Scheduler) does,
29 `yield()` function which suspends a task is not implemented,
30 `startLoop()` function which allocates a stack to the new task is not implemented.
31 
32 **jm_Scheduler** schedules tasks sequentially on the stack processor.
33 The rules to _yield_ and _resume_ are:
34 
35 - _yield_ comes out when routine leaves at end of function or by an explicit `return` instruction.
36 - _resume_ to a next state can be done with a variable and a `switch` instruction. Or:
37 - _resume_ to a next state can be done by switching to another function.
38 - Persistent variables must be implemented _global_ or _local_ with the pragma `static`.

76 ### Timestamp
77 
78 The _timestamp_ is read from the Arduino function `micros()`.
79 By design, the `micros()` function of Arduino UNO and Leonardo running at 16MHz returns a [us] _timestamp_ with a resolution of [4us].
80 
81 `micros()` declaration is:
82 
83 ```C
84 unsigned long micros();
85 ```
86 
87 Look at https://www.arduino.cc/en/Reference/Micros for details.

101 ### Timestamp declaration and constants
102 
103 ```C
104 typedef uint32_t timestamp_t;
105 
106 #define timestamp_read() ((timestamp_t)micros())
107 
108 #define TIMESTAMP_DEAD (0x01CA0000) // routine dead time [30s + 15ms + 488us]
109 #define TIMESTAMP_TMAX (0xFE35FFFF) // [1h + 11m + 4s + 951ms + 808us - 1]
110 
111 #define TIMESTAMP_1US (1UL) // [1us]
112 #define TIMESTAMP_1MS (1000*TIMESTAMP_1US) // [1ms]
113 #define TIMESTAMP_1SEC (1000*TIMESTAMP_1MS) // [1s]
114 #define TIMESTAMP_1MIN (60*TIMESTAMP_1SEC) // [1 minute]
115 #define TIMESTAMP_1HOUR (60*TIMESTAMP_1MIN) // [1 hour]
116 ```
117 
118 > `timestamp_t` defines the type of all _timestamp_ values.
119 
120 > `timestamp_read()` returns the instantaneous _timestamp_.
121 This function can also be used by interrupt routines to _timestamp_ they data.
122 
123 > `TIMESTAMP_DEAD` is the maximum allowed execution time of a routine to guarantee right scheduling.
124 If the routine doesn't end before, the scheduler could miss very long scheduling (see next).
125 
126 > `TIMESTAMP_TMAX` is the maximum allowed scheduling time of a routine.
127 In practice, don't use _timestamp_ values greater than 1 hour.

129 ### jm_Scheduler methods
130 
131 ```C
132 // start routine immediately
133 void start(voidfuncptr_t func);
134 
135 // start routine immediately and repeat it at fixed interval
136 void start(voidfuncptr_t func, timestamp_t ival);
137 
138 // start routine on time and repeat it at fixed interval
139 void start(voidfuncptr_t func, timestamp_t time, timestamp_t ival);
140 
141 // stop routine, current or scheduled, remove it from chain
142 void stop();
143 
144 // rearm current routine and set or reset interval
145 void rearm(timestamp_t ival);
146 
147 // rearm current routine, change routine function and set or reset interval
148 void rearm(voidfuncptr_t func, timestamp_t ival);
149 ```
150 
151 > `start()` initiates a scheduler variable, starts a routine function, immediately or on time, with or without repetitions.
152 `start()` is invoked once. Next `rearm()` allows changing scheduler values.
153 
154 > `stop()` cancels further execution of a scheduled routine.
155 `stop()` can be invoked from inside routine or elsewhere.
156 If invoked from inside _routine_, `stop()` doesn't exit the function, just cancels further execution.
157 
158 > `rearm()` changes values of a scheduler variable.
159 The new values are evaluated on exit routine function.
160 The main usage is to change _interval_ or _function_ or both or else cancel further execution.

163 ### jm_Scheduler loop
164 
165 ```C
166 static void cycle();
167 ```
168 
169 > `cycle()` is the cornerstone of the scheduler and must be invoked as often as possible.
170 Note that `cycle()` is a _static_ _method_.
171 The right place is in Arduino `loop()` function.
172 Example:
173 
174 ```C
175 void loop(void)
176 {
177  jm_Scheduler::cycle();
178 }
179 ```
180 
181 > `cycle()` can also be invoked in Arduino `setup()` function. Example:
182 
183 ```C
184 void setup(void)
185 {
186  // here, some jm_Scheduler variables initialized...
187 
188  Serial.begin(9600);
189  while (!Serial)
190  {
191  // wait for USB Serial ready...
192 
193  jm_Scheduler::cycle();
194  }
195 
196  // split long setup()...
197 
198  jm_Scheduler::cycle();
199 
200  // continue setup()...
201 }
202 ```
203 
204 > `cycle()` can't be invoked from inside a routine function.

207 ### Good scheduling practices
208 
209 - To guarantee a good scheduling of all tasks,
210 the execution time of each function must be as short as possible.
211 
212 - Avoid Arduino `delay()` function, use **jm_Scheduler** `rearm()` method with appropriate arguments to split the routine in some serialized functions.
213 
214 - Use same technical for long calculations.

219 Here are some hacks that can be implemented by modifying the file **jm_Scheduler.h**.
